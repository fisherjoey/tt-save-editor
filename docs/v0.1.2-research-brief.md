# v0.1.2 research brief — UE 5.6 GVAS StructProperty tag format

**The blocker:** to safely allow byte-length-changing enum edits (the v0.1.1 restriction), the editor needs to walk the GVAS body recursively and update every container's outer `Size` field when an inner value grows or shrinks. The walker stalls on `StructProperty` tags because UE 5.5+ changed the format and we haven't decoded the new layout reliably enough to ship.

This document is the working state of what we know, what we don't, and what the next session's research should focus on.

## What the editor needs

Given an edit at byte offset `O` that grows the body by `Δ` bytes, find every container (`StructProperty` / `ArrayProperty` / `MapProperty`) whose value range contains `O`, and update each one's `Size` int32 by `Δ`. Outermost first so offsets stay valid.

That requires a structural parser that walks the body recursively. The scalar scanner we already have (`packages/core/src/gvas/scan.ts`) works by forward-scanning for tag signatures and is good enough for finding editable values, but it doesn't track parent containers.

We started one in [`packages/core/src/gvas/structure.ts`](../packages/core/src/gvas/structure.ts). It handles the legacy `StructProperty` tag layout but fails on UE 5.5+ saves like Dinner.

## The format question

For a normal property in this game's saves, the tag is:

```
name        FString
type        FString
arrayIndex  int32
size        int32
(type-specific bytes — empty for most scalars)
hasGuid     uint8
guid        16 bytes if hasGuid
value       size bytes
```

That's been verified against real bytes. **The wrinkle is `StructProperty`.** In UE 5.5+ (which Dinner uses — engine branch `++Dinner+mainline`, 5.6.1), the type-specific bytes use `TopLevelAssetPath` — separate `AssetName` and `PackageName` FNames — instead of the legacy `structName` FString + 16-byte `structGuid`.

### Evidence from a real save

Real bytes for the `SaveId` StructProperty (file offset starting at 2347, body offset 54):

```
2347  07 00 00 00 53 61 76 65 49 64 00              | "SaveId\0"  (name FString, length 7)
2358  0f 00 00 00 53 74 72 75 63 74 50 72 6f 70 65  | "StructProperty\0"  (type FString, length 15)
      72 74 79 00
2377  01 00 00 00                                   | int32 = 1   (arrayIndex)
2381  05 00 00 00                                   | int32 = 5   ← if this is `size`, value is 5 bytes, doesn't match the 16-byte Guid that follows
2385  47 75 69 64 00                                | "Guid"  (length-prefixed??? or raw?)
2390  01 00 00 00                                   | int32 = 1
2394  14 00 00 00                                   | int32 = 20
2398  2f 53 63 72 69 70 74 2f 43 6f 72 65 55 4f 62  | "/Script/CoreUObject\0"  (FString, length 20)
      6a 65 63 74 00
2418  00 00 00 00                                   | 4 bytes of 0
2422  10 00 00 00                                   | int32 = 16  ← this DOES match the 16-byte Guid that follows
2426  08                                            | 1 byte = 0x08 (?? hasGuid is 0/1; this doesn't fit)
2427  b4 c2 b4 59 95 0d 7a 4b a1 af 90 8e 29 03 c9  | 16 bytes (a GUID)
      a9
```

Best-effort interpretation (unverified):

```
arrayIndex int32              = 1
structName/AssetName FString  = "Guid"  (length 5 = 4 chars + NUL)
something int32               = 1     (??)
PackageName FString           = "/Script/CoreUObject"
something int32               = 0     (??)
size int32                    = 16
hasGuid uint8 or padding      = 0x08  (??)
value (= the 16-byte Guid)
```

**Open questions:**

1. What are the two mystery int32s at offsets 2390 and 2418? Both look like length-or-flag fields. `1` and `0` for SaveId; values may differ for other structs.
2. Is the `0x08` at offset 2426 actually `hasGuid`? In other property types we've seen `hasGuid` is always 0 or 1. `0x08` doesn't fit. Possibly it's a different field, or `hasGuid` lives elsewhere.
3. Is `AssetName` always 5 bytes here ("Guid" + NUL), or does it come from a generic FName/FString?
4. Does this format also apply to **nested** StructProperties inside `ArrayProperty<StructProperty>` and `MapProperty` values, or only top-level ones?
5. What about the inline element-tag block that an `ArrayProperty<StructProperty>` writes once before its elements — does it use the same UE 5.5+ layout?

### Other helpful examples to decode

Multiple `SavedGameProgressEnumValues` arrays exist (each is `ArrayProperty<StructProperty>` containing `TtGameProgressSavedEnumValue` elements). The byte structure of one of these arrays (specifically, the inline `StructProperty` element-type tag that follows the count) is the other place we need to handle.

The `MapProperty` for currencies (`Currencies` map with `DinnerCurrency_Studs`, `DinnerCurrency_WayneTechChips`) is another container we'd want to verify.

## What we already know

- **RC4 cipher with 32-byte key** — credit @RealDarkCraft. The key is in [`packages/core/src/crypt/index.ts`](../packages/core/src/crypt/index.ts).
- The GVAS **header** (engine version, custom-version array, save-class name) parses cleanly. See [`packages/core/src/gvas/header.ts`](../packages/core/src/gvas/header.ts).
- The **scalar scanner** finds editable fields by forward byte signature. It works because it doesn't try to model containers. See [`packages/core/src/gvas/scan.ts`](../packages/core/src/gvas/scan.ts).
- Property tag order is `name, type, arrayIndex, size, …` (verified empirically — earlier UE docs sometimes show the opposite order; for this game it's arrayIndex first).
- For scalar / Bool / FString-payload properties, tag layout works as documented. The break is at `StructProperty` and we suspect `ObjectProperty` / `SoftObjectProperty` use the same UE 5.5+ TopLevelAssetPath pattern.

## What success looks like for v0.1.2

A function `parseStructure(body) → tree` such that:

1. Walking the whole body produces a tree with N top-level properties (matches the count from forward-scanning the same body).
2. Every container node has accurate `sizeOffset`, `valueStart`, `valueEnd` fields.
3. Given any enum value's byte offset, `ancestorContainers(tree, offset)` returns the chain of containers that contain it.
4. After an edit that grows the body by Δ, updating every ancestor's `Size` field by Δ and re-walking the new body returns the same tree shape (no garbled subsequent properties).

The acceptance test for v0.1.2: **change a `Locked → Collected` enum (Δ=3 bytes), load it in the pre-patch game, and confirm both the targeted entry AND every neighbouring entry retains its prior state.** That's the actual user-facing bug we're fixing.

## References worth checking

- **UE 5.5 source — `FPropertyTag::SerializeType()`** in `Engine/Source/Runtime/CoreUObject/Private/UObject/PropertyTag.cpp`. The most reliable source for the new tag format. UE 5.5 release notes also mentioned the TopLevelAssetPath change.
- **[trumank/uesave](https://github.com/trumank/uesave)** — Rust GVAS parser. Issue #49 specifically mentions encrypted UE5 saves, but the underlying parser may handle the new StructProperty tag. Look at how its `PropertyTag::read` handles `StructProperty` in 5.5+.
- **[SparkyTD/UnrealEngine.Gvas](https://github.com/SparkyTD/UnrealEngine.Gvas)** — .NET GVAS parser.
- **[AcK77/TTGames-Explorer-Rebirth](https://github.com/AcK77/TTGames-Explorer-Rebirth)** — TT Games asset toolkit (different domain, but useful for TT-specific quirks).

## How to verify against real bytes

Reference saves live in `packages/core/test/fixtures/`. The minimal `slot1_prepatch.sav` (~60 KB) is committed; larger / third-party saves are gitignored.

To dump bytes of a property:

```bash
cd packages/core
cat > /tmp/i.ts <<'JS'
import { readFileSync } from "node:fs";
import { decrypt } from "./src/crypt/index.js";
import { parse } from "./src/gvas/index.js";
const plain = parse(decrypt(new Uint8Array(readFileSync(process.argv[2])))).body;
// dump hex+ascii of a byte range around a known offset
function dump(off, n) {
  for (let o = off; o < off + n; o += 16) {
    const hex = Array.from(plain.slice(o, o + 16)).map(b => b.toString(16).padStart(2, "0")).join(" ");
    const asc = Array.from(plain.slice(o, o + 16)).map(b => (b >= 32 && b < 127) ? String.fromCharCode(b) : ".").join("");
    console.log(`${o.toString().padStart(6)}  ${hex.padEnd(48)}  |${asc}|`);
  }
}
dump(40, 200); // SaveName + SaveId region
JS
npx --yes tsx /tmp/i.ts test/fixtures/slot1_prepatch.sav
```
